# C3 — Manual release pipeline (semantic version bump + GitHub Release)

A **manually-triggered** GitHub Actions workflow (`workflow_dispatch`) releases the
plugin. Its single input is the **release type** — `major`, `minor`, or
`maintenance` — and the job derives the next [semantic version](https://semver.org)
from the current `pluginVersion`:

- `major` → `X+1.0.0`
- `minor` → `X.Y+1.0`
- `maintenance` → `X.Y.Z+1` (patch / bug-fix)

The job: runs the regular test suite (must pass before anything is released);
computes and writes the next `pluginVersion`; builds the plugin distribution zip;
generates a changelog from the commits since the previous tag; commits the version
bump; creates an annotated git tag `vX.Y.Z`; and publishes a **GitHub Release** for
that tag whose **assets are the plugin distribution zip and the changelog**.

Implementation may use plain GitHub Actions + the `gh` CLI, or a release framework
such as JReleaser — whichever is simplest; the behaviour above is what matters.

## Acceptance Criteria

- AC-C3-01: A GitHub Actions workflow is **manually triggered** (`workflow_dispatch`)
  and takes a single `release_type` choice input with values `major`, `minor`,
  `maintenance`.
- AC-C3-02: The job computes the next version from the current `pluginVersion`
  according to the chosen release type (major/minor/patch bump) and updates
  `pluginVersion`.
- AC-C3-03: The regular test suite runs and must pass before the release is
  created.
- AC-C3-04: The job creates a git tag `vX.Y.Z` and a corresponding GitHub Release.
- AC-C3-05: The GitHub Release's assets include both the plugin distribution zip
  and the generated changelog.

## Variant — Claude-driven release (`release-using-claude.yml`)

An alternative, separately-dispatchable workflow (job `release-using-claude`)
produces the **same release outcome** as above — tests must pass, the plugin is
built at the chosen version via `-PpluginVersion`, an annotated `vX.Y.Z` tag is
created, and a GitHub Release is published whose assets are the distribution zip
and the changelog — but the **next version and the changelog are generated by
Claude** (via the `/release-notes` skill) instead of the deterministic bash bump:

- It takes **no `release_type` input**. Claude reads the commits since the latest
  `v*` tag and picks the next best semantic version per the skill's rules
  (`BREAKING CHANGE:` → major, any `feat:` → minor, only `fix:`/`perf:` → patch),
  then writes the bare `X.Y.Z` to `RELEASE_VERSION.txt` and the notes to
  `CHANGELOG.md`. A subsequent step validates the version format and that the tag
  does not already exist before tagging/releasing.
- Like the primary workflow, it never pushes to `main`; the tag remains the source
  of truth for the released version.
- The original `release.yml` (deterministic bump, AC-C3-01…05) remains the default
  and is unchanged.
